# research

The replication material behind falsegreen's validation and the empirical work it
feeds. This directory is the data and the scripts; the per-project decisions live in
[../VALIDATION.md](../VALIDATION.md) and the open issues track the analysis.

## What is here

- **`collect.py`** - a reproducible harness. Given a directory of cloned projects, it
  runs the scanner on each and writes one CSV row per project (test count, total
  findings, HIGH count, web/browser-layer count, and a count per detection code). It
  is re-runnable and idempotent per project.

  ```bash
  PYTHONPATH=../src python collect.py <projects_root> data/dataset-python.csv
  ```

- **`data/dataset-python.csv`** - the seed dataset: findings on eight public Python
  projects (httpx, starlette, flask, fastapi, django-rest-framework, aiohttp, sanic,
  werkzeug), each over 200 stars with 500 or more test functions. Raw findings, not
  yet interpreted; the false-positive analysis that came out of it is in the issues
  (a C7 reflexivity false positive, a C4 route-handler false positive).

- **`PROTOCOL.md`** - the data-collection design. Two datasets: an observational one
  (the scanner on real projects, started here) and a controlled one (unit tests
  generated by several models and then labeled by hand, the ground truth for studying
  AI-written false-positive tests).

## How collection works

Collection is session-driven, not autonomous: a project is added when it is run and
appended, with the decision recorded in `VALIDATION.md`. Cross-language coverage of the
semantic pass is evaluated separately (the scanner is Python-only by design).

## Honesty note

The scanner findings in the dataset are raw. A HIGH count on a project does not mean
the project has a bug; it may be a false positive in the tool, which is the point of
running it against real code. Each one is triaged in the issues before any claim is
made.
